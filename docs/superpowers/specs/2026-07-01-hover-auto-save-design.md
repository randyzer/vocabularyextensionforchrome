# 悬停查词自动保存优化设计

- 日期：2026-07-01
- 状态：已确认，待实施

## 目标

在保留“悬停即可收集单词”的低操作成本前提下，降低短暂停留造成的误保存，并避免用户在同一页面反复悬停同一处文本时重复触发保存。

## 交互流程

1. 鼠标在同一单词目标上稳定停留 500ms 后发起查词。
2. 查词成功后开始 2000ms 自动保存倒计时。
3. 倒计时期间目标始终未变化，才触发保存。
4. 保存成功后显示现有的“已保存”状态和 5 秒撤销入口。

从首次悬停到保存，最短等待约为 2500ms，另加实际查词耗时。

以下任一事件发生时，取消尚未完成的查词或保存倒计时：

- 鼠标移动到另一个文本目标或空白区域；
- 页面滚动；
- 浏览器窗口或当前页面失去焦点。

## 页面级防重

内容脚本为当前页面维护一个内存集合 `savedTargetKeys`。目标键由以下信息组成：

- 单词；
- 规范化后的上下文句子；
- 单词在句子中的起始位置。

同一目标在当前页面保存成功后：

- 再次悬停仍可查词并展示释义；
- 不再调用保存接口；
- 不增加数据库中的遇见次数。

该集合只在当前页面生命周期内有效。刷新、重新导航或打开新页面后自然清空，允许用户在新的阅读会话中再次记录遇见。

数据库现有的“词形 + 规范化句子 + 来源站点”去重逻辑继续保留，作为后端一致性保护，不需要修改数据结构。

## 失败与撤销语义

- 只有保存接口明确返回成功后，才把目标键加入 `savedTargetKeys`。
- 保存失败时不加入集合；用户再次稳定悬停时可以重试。
- 撤销接口明确返回成功后，从集合中移除该目标键；随后再次悬停可重新保存。
- 撤销失败时保留目标键，避免前端状态与数据库中的已保存记录相互矛盾。

## 预计修改范围

- `src/shared/constants.ts`：将查词成功后的自动保存等待时间调整为 2000ms。
- `src/content/hover-controller.ts`：沿用现有计时与取消机制。
- `src/content/index.ts`：增加当前页面的目标防重、成功标记及撤销解除逻辑。
- `tests/unit/hover-controller.test.ts`：更新计时行为测试。
- 内容脚本相关单元测试：补充页面级防重、失败重试和撤销后重存覆盖。

## 测试场景

1. 悬停 500ms 时查词，查词成功后再等待 2000ms 才保存。
2. 保存倒计时内目标变化、滚动或失焦时不保存。
3. 同一目标在当前页面保存成功后，重复悬停不会再次保存。
4. 重复悬停已保存目标时仍能展示查词结果。
5. 保存失败后，再次悬停可以重试保存。
6. 撤销成功后，再次悬停可以重新保存。
7. 撤销失败后，当前页面仍阻止重复保存。
8. 刷新或进入新页面后，页面级防重状态重置。

## 非目标

- 不改为点击确认保存。
- 不跨页面或跨浏览器会话持久化防重状态。
- 不修改数据库表结构或已有后端去重规则。
- 不改变查词目标识别规则、词典来源或复习摘要逻辑。

## 验收标准

- 查词成功后，鼠标保持在同一目标 2000ms 才自动保存。
- 同一页面、同一文本位置最多成功保存一次，除非用户先成功撤销。
- 失败操作不会造成无法重试或前后端状态不一致。
- 现有查词、提示、撤销和页面事件取消行为无回归。
